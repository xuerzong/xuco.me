---
title: 流式输出的 Markdown 渲染优化
description: 优化 AI 流式输出场景下的 Markdown 渲染性能。
---

在 AI 应用里，流式输出（streaming output）几乎已经成了默认选项。用户不用等模型把整段内容都生成完，界面就可以先把正在生成的部分展示出来，整体体验会自然很多。

但在渲染流式输出的 Markdown 内容时，会发现一个问题：内容虽是“逐字追加”，渲染却是“整块渲染”。

这就会出现几个很典型的问题：

- **页面闪烁**：每次 token 更新都触发整段 Markdown 重新渲染，视觉上会有明显抖动。
- **性能下降**：内容越长，重新解析和重新渲染的成本就越高。
- **结构不稳定**：代码块、表格、公式这类结构化内容在未闭合时很容易渲染异常。

问题的根源并不复杂：**我们把“流式追加文本”当成了“重新渲染整篇文档”来处理。**

## 一个更合适的思路：分块渲染

为了优化上述问题，我们可以不让 Markdown 组件每一次都从头解析整段字符串，而是先把流式文本拆成一组相对稳定的内容块，再分别渲染。

核心原则只有两条：

- 已经确认完成的块不再重复解析，也不再重复渲染。
- 新增内容只影响最后一个块，或者追加新的块。

这样一来，渲染的成本就会从“整篇文档重复处理”变成“只处理增量部分”，页面也会稳定很多。

具体实现思路可以参考 [json-render](https://json-render.dev/)。本质上，它做的事情并不神秘，就是先把 Markdown 转成一层更适合前端增量更新的中间结构。

## 先定义一个中间结构

参考 [json-render](https://json-render.dev/)， 可以先把 Markdown 解析成下面这样的数据：

```typescript
type MarkdownBlock = {
  type: 'codeblock' | 'formula' | 'table' | 'list' | 'paragraph' | 'heading'
  content: string | string[] | MarkdownBlock[]
}

type MarkdownContent = MarkdownBlock[]
```

这里的重点不是“重新发明 Markdown AST”，而是给 UI 一种更适合流式场景的数据形态。

对于渲染层来说，它未必需要一棵完整的语法树，它真正关心的通常只有三件事：

- 当前这段内容是什么类型；
- 它是否已经稳定；
- 下一次更新时应该替换它，还是在后面追加。

## 一个最小示例

假设模型正在输出下面这段 Markdown：

````markdown
This is a paragraph.

```js
console.log('hello world')
```

| Name  | Age |
| ----- | --- |
| Alice | 30  |

$$E=mc^2$$
````

解析之后，可以先变成一组明确的内容块：

````json
[
  {
    "type": "paragraph",
    "content": "This is a paragraph."
  },
  {
    "type": "codeblock",
    "content": "```js\nconsole.log('hello world')\n```"
  },
  {
    "type": "table",
    "content": "| Name  | Age |\n| ----- | --- |\n| Alice | 30  |"
  },
  {
    "type": "formula",
    "content": "$$E=mc^2$$"
  }
]
````

一旦变成这种结构，前端的处理方式就会简单很多。

段落、代码块、表格、公式都可以作为独立节点存在，而不是让一个越来越长的 Markdown 字符串反复进入同一套解析流程。

## 流式场景真正难的地方：不完整内容

**“渲染一个还没结束的 Markdown”** 会造成结构不稳定的渲染结果。

比如模型可能会先返回：

````markdown
```ts
function sum(a: number, b: number) {
````

这时候代码块还没有闭合。对于常规 Markdown 渲染器来说，这往往意味着：

- 要么整块解析失败；
- 要么被当成普通文本；
- 要么下一次 token 到来时整段重新纠正。

如果每次都依赖完整 Markdown 重新解析，那么抖动几乎是必然的。

所以这里需要再加一个维度：**块是否完整。**

```typescript
type MarkdownBlock = {
  type: 'codeblock' | 'formula' | 'table' | 'list' | 'paragraph' | 'heading'
  content: string | string[] | MarkdownBlock[]
  isComplete: boolean
}

type MarkdownContent = MarkdownBlock[]
```

有了 `isComplete` 之后，渲染层的处理就会稳定很多：

- `isComplete: true` 的块直接按正式样式渲染；
- `isComplete: false` 的块先用占位样式或降级样式渲染；
- 当块闭合后，只更新这一个块的状态，而不是重算整篇内容。

## 如何判断一个块是否完整

这件事不一定要做得很重。很多场景下，用一套轻量规则就已经够了：

- 代码块：是否遇到了成对的 ````` 分隔符；
- 行内或块级公式：分隔符是否闭合；
- 表格：表头和分隔行是否已经形成；
- 列表：是否仍在连续列表上下文中；
- 段落：通常可以认为在遇到空行或新块类型时完成。

换句话说，流式解析不一定非要一步到位构建完整 AST，它更像是一个小型状态机：

- 当前是否处于代码块内部；
- 当前是否处于公式块内部；
- 当前缓冲区应该归属于哪个块；
- 新 token 到来时，是更新当前块，还是提交一个完成块。

## 一个更贴近实现的处理流程

如果沿着前端渲染链路往下拆，整个过程大致可以分成这几步：

1. 接收模型流式输出的原始文本。
2. 把新增文本喂给一个增量解析器，而不是重新处理全文。
3. 增量解析器输出 `MarkdownContent`。
4. 对于已经完成的块，直接复用之前的渲染结果。
5. 对于最后一个未完成块，允许它被持续更新。

这里最关键的，不是“解析得有多精准”，而是“更新边界是否足够稳定”。

只要前面的内容块不会频繁回退，性能和视觉稳定性通常都会有很明显的改善。

比如一篇已经生成了 200 行内容的回答，只新增最后 10 个字符（最后一行）时：

- 原始方案：重跑 200 行内容的解析与渲染；
- 分块方案：只更新最后一个块，前面 199 行保持不动。

内容越长、结构越复杂，这种差异就会越明显。

## 总结

流式 Markdown 渲染真正的问题，并不是“Markdown 很难渲染”，而是“不该在流式场景里反复重渲染完整 Markdown”。

通过将 Markdown 内容拆成一组相对独立的块，并跟踪每个块的状态，我们就可以大幅降低流式输出场景下的渲染成本和视觉抖动。
